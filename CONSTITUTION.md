# moduleBox — конституция

> **Статус:** фундамент — незыблемые правила схемы.
>
> Этот документ описывает, как контроллеры moduleBox общаются с внешним миром через UART, UDP, OSC, MQTT.
> Правила ниже — стабильное ядро. Они одинаковы для всех модулей и не должны меняться
> без веской причины: на них опираются прошивка, конфигуратор и сторонние интеграции.
>
> Актуальный набор модулей и их событий/команд для конкретной прошивки берётся из
> сгенерированного `manifest.json` — он и есть живой реестр возможностей. Код обязан
> соответствовать этой конституции; отдельной рукописной таблицы по модулям нет,
> чтобы исключить расхождения.

---

## 1. Идея в одном абзаце

Контроллер состоит из **модулей**. Каждый модуль либо что-то **сообщает** миру
(нажали кнопку, кончился трек, изменилось значение датчика), либо **принимает**
команды (зажги лампу, играй трек, усни). Первое — `event`, второе — `action`.
Больше направлений нет. Та же пара `event`/`action` используется и для внутренних
кросслинков (модуль → модуль), и для связи с внешним брокером — язык один.

---

## 2. Структура топика

```
<deviceName>/<module>_<slot>/<direction>/<name>
```

| Часть          | Что это                                   | Пример          |
|----------------|-------------------------------------------|-----------------|
| `deviceName`   | Имя устройства, задаёт пользователь       | `hall-panel`    |
| `module`       | Имя модуля (одинаково для родственных)    | `buttonLed`     |
| `slot`         | Номер слота, отделён `_`                   | `2`             |
| `direction`    | Направление: `event` или `action`         | `event`         |
| `name`         | Что именно произошло / что сделать         | `longPress`     |

Полный пример:

```
hall-panel/buttonLed_2/event/longPress     payload: 1
hall-panel/buttonLed_2/action/led          payload: 255 255 255
```

**Минимум четыре уровня после `deviceName`.**
`name` обязателен всегда — односложных модулей без имени нет. `name` может
содержать дополнительные сегменты (`ch_0`, `ch_1` и т.п.) — глубина растёт
вправо от `name`, а не вставляется между фиксированными уровнями. Это позволяет
прошивке разбирать топик без ветвлений: первые три уровня читаются по позиции.

---

## 3. Направления

| Направление | Кто пишет           | Смысл                                  |
|-------------|---------------------|----------------------------------------|
| `event`     | модуль (контроллер) | «случилось вот это» — модуль → мир      |
| `action`    | мир / кросслинк     | «сделай вот это» — мир → модуль         |

Внешняя система **подписывается** на `event/*` и **публикует** в `action/*`.
Кросслинк делает то же самое внутри контроллера: слушает чей-то `event/*`
и пишет в чей-то `action/*`. Поэтому одна автоматизация работает одинаково
и локально, и через брокер.

---

## 4. Payload

Полезная нагрузка — **голое значение**, без обёрток и JSON.

| Тип      | Формат                  | Пример          |
|----------|-------------------------|-----------------|
| bool     | `0` / `1`               | `1`             |
| int      | целое в виде строки     | `80`            |
| float    | дробное в виде строки   | `3.14`          |
| string   | строка                  | `flash`         |
| rgb      | три числа через пробел  | `255 255 255`   |

Имя величины всегда в **топике**, значение — в **payload**. Двоеточие и прочие
разделители «имя:значение» внутри одной строки не используются.

> Записи вида `longPress:1` или `ledMode:flash`, встречающиеся в обсуждениях, —
> это человеческая нотация «топик `<имя>` со значением `<value>`». В реальном
> MQTT имя и значение разнесены:
>
> ```
> topic:   buttonSmartLed_3/action/ledMode
> payload: flash
> ```

Так `<deviceName>/+/event/longPress` ловит все долгие нажатия в системе
независимо от значения, а `0`/`1` различаются по payload.

---

## 5. Retain (флаг хранения)

Брокер может хранить последнее значение топика и отдавать его при подключении.
Правило:

| Что                       | retain | Почему                                            |
|---------------------------|:------:|---------------------------------------------------|
| `event/enable`            |   да   | Состояние «жив/спит» нужно знать сразу при connect |
| Прочие `event/*`          |  нет   | Мгновенный факт; старый клик не должен «вспоминаться» |
| `action/*`                |  нет   | Команда исполняется один раз, не хранится          |

Состояния-флаги, которые осмысленно читать в любой момент (а не только в момент
изменения), могут публиковаться с `retain = да` — но это решается для каждого
имени в его описании, по умолчанию `event/*` не retained.

---

## 6. Зарезервированное имя `enable`

У каждого модуля есть жизненный цикл: активен или спит. Это не отдельная ветка
топиков, а зарезервированное имя внутри обычной пары `event`/`action`:

```
<dev>/<module>_<slot>/event/enable     1   (retained) — рапорт: я активен, со мной можно работать
<dev>/<module>_<slot>/event/enable     0   (retained) — рапорт: я сплю
<dev>/<module>_<slot>/action/enable    1              — команда: проснись
<dev>/<module>_<slot>/action/enable    0              — команда: усни
```

По умолчанию все модули активны на старте. Флаг конфигурации `disableOnStart`
оставляет модуль спящим до явной команды `action/enable 1`.

Сценарий «кросслинк разбудил модуль — мир должен узнать»: команда приходит в
`action/enable 1`, модуль просыпается и публикует `event/enable 1` (retained).
Подписчик на `<dev>/+/event/enable` мгновенно видит, с кем можно работать.

---

## 7. Правила именования

### deviceName
Используется и в MQTT-топике, и в mDNS (имя хоста), поэтому ограничен жёстче:

- разрешены: строчные латинские буквы, цифры, дефис `-`
- запрещены: подчёркивание `_`, точка, пробел, спецсимволы
- примеры: `hall-panel`, `mb1`, `stage-left`

`_` запрещён только в `deviceName` (из-за mDNS). В номерах модулей `_` —
обычный разделитель, в mDNS он не попадает.

### module_slot
Генерируется прошивкой, пользователь не задаёт вручную:

- `<имяМодуля>` + `_` + `<номерСлота>`
- имя модуля — в стиле camelCase, как в манифесте (`buttonLed`, `mp3Player`)
- родственные модули носят одно имя в топике ради преемственности
  (например `mp3Player` и `wavPlayer` могут оба публиковаться как `player`)

### name (событие/действие)
- короткий глагол или существительное в camelCase: `press`, `longPress`, `setVal`, `play`
- без пунктуации, скобок и спецсимволов
- одно и то же явление называется одинаково во всех модулях (см. §8)

## 8. Канонический словарь имён

Чтобы родственные модули не плодили синонимы (`press` / `pressed` / `btn`),
новые компоненты берут имена отсюда. Список расширяется по мере надобности,
но существующие имена не переименовываются без причины.

### Общие (любой модуль)
| Имя       | Направление | Тип  | Смысл                              |
|-----------|-------------|------|------------------------------------|
| `enable`  | event+action| bool | Активен / спит                     |

### Кнопки / ввод
| Имя         | Направление | Тип  | Смысл                            |
|-------------|-------------|------|----------------------------------|
| `press`     | event       | bool | Нажатие (1) / отпускание (0)     |
| `longPress` | event       | bool | Долгое нажатие (1) / отпускание (0) |
| `click`     | event       | bool | Одиночный клик                   |
| `doubleClick`| event      | bool | Двойной клик                     |

### Значения / датчики
| Имя        | Направление  | Тип       | Смысл                          |
|------------|--------------|-----------|--------------------------------|
| `val`      | event        | int/float | Текущее значение               |
| `setVal`   | action       | int/float | Задать значение                |
| `threshold`| event        | bool      | Порог пройден                  |

### Свет / индикация
| Имя      | Направление | Тип | Смысл                |
|----------|-------------|-----|----------------------|
| `led`    | action      | rgb | Цвет подсветки       |
| `bright` | action      | int | Яркость 0–100        |

### Воспроизведение
| Имя             | Направление | Тип  | Смысл                  |
|-----------------|-------------|------|------------------------|
| `play`          | action      | int  | Играть трек номер N    |
| `stop`          | action      | bool | Остановить             |
| `setVolume`     | action      | int  | Громкость 0–100        |
| `endOfTrack`    | event       | bool | Трек закончился        |

> Таблицы выше — стартовый словарь канонических имён, не полный перечень. Точный
> набор событий и команд каждого модуля в конкретной прошивке см. в `manifest.json`.

---

## 9. Что НЕ входит в схему

- **Телеметрия не образует отдельной ветки.** Периодический рапорт идёт в тот
  же `event/<name>`, что и рапорт по изменению. «Периодичность» — это свойство
  модуля (`periodic`, `refreshRate`, `statusPeriod`), а не отдельный топик.
- **Конфигурация не хранится в MQTT.** Параметры модуля (`volume`, `inverse`,
  `debounceGap` …) задаются манифестом, а не топиками. MQTT — это рантайм-обмен
  событиями и командами, не база настроек.

---

## 10. Сводка одной картинкой

```
hall-panel/buttonLed_2/event/press      1            кнопку нажали
hall-panel/buttonLed_2/event/longPress  1            держат долго
hall-panel/buttonLed_2/event/enable     1   retain   модуль активен
hall-panel/buttonLed_2/action/led       0 128 255    зажги синим
hall-panel/buttonLed_2/action/bright    80           яркость 80%
hall-panel/buttonLed_2/action/enable    0            усни

hall-panel/player_0/event/end           1            трек кончился
hall-panel/player_0/action/play         5            играй трек 5
hall-panel/player_0/action/volume       70           громкость 70

hall-panel/buttonSmartLed_3/action/ledMode  flash     режим подсветки (строка)

hall-panel/relay_3/event/val            1   retain   реле включено
hall-panel/relay_3/action/setVal        0            выключи реле

hall-panel/pwmLeds_2/event/enable            1   retain   модуль активен
hall-panel/pwmLeds_2/action/setRGB           255 128 0    установить цвет (R G B)
hall-panel/pwmLeds_2/action/ch_0/setBright   128          яркость канала 0
hall-panel/pwmLeds_2/action/ch_1/setBright   200          яркость канала 1
hall-panel/pwmLeds_2/action/ch_2/setBright   64           яркость канала 2
```

> `ch_0/setBright` — пример многосегментного `name`: канал указан как префикс,
> действие — как суффикс. Парсер читает первые три уровня по позиции
> (`deviceName` / `module_slot` / `direction`), остальное — `name` целиком.
